## Product Scope

### MVP - Minimum Viable Product

**核心功能（必须能工作才有用）：**

1. **Excel 上传和自动 key 生成**
   - 支持 12 种语言（CN、DA、DE、EN、ES、FI、FR、IT、NL、NO、PL、SE）
   - 自动生成唯一 UUID 作为每条文案的 key
   - 支持按项目上传和管理多语言文案
   - 自动识别 Excel 中的新增和修改内容，更新数据库
   - Excel 解析成功率：≥ 95%

2. **三端格式转换**
   - 支持 iOS（.strings）、Android（strings.xml）、Web（JSON）三种格式
   - 文件格式转换准确性：≥ 99.9%
   - 下载的文件格式符合各平台标准，可直接使用

3. **按需下载**
   - 支持按项目、语言、平台筛选下载特定的多语言文件
   - API 响应时间（1000 条文案）：< 10 秒
   - 同时支持脚本下载（通过 API）和手动下载（通过 Web 界面）

**MVP 成功标准：**

- 至少两个项目成功使用 MVP
- 产品经理能够在 10 分钟内完成 100 条文案（10种语言）的 Excel 上传，包括预览和差异确认
- 开发人员能够在 30 秒内拉取三端语言包（30个文件）
- 两个项目的三端文案都能保持一致，不再出现 key 不一致的问题
- 系统可用性：≥ 99%

### Growth Features (Post-MVP)

**功能增强（让产品具有竞争力）：**

1. **Web 界面单条编辑**
   - 手动编辑、删除、查询单条文案
   - 支持在 Web 界面中直接操作单条文案
   - 提供灵活的文案管理方式

2. **文案搜索和筛选**
   - 按关键、文案内容、语言等条件搜索
   - 按项目、语言等维度筛选文案
   - 快速定位和管理文案

3. **变更历史和版本回滚**
   - 记录每条文案的变更历史
   - 支持版本回滚功能
   - 提供版本对比和预览

4. **API + 脚本支持**
   - 完整的 API 文档（Swagger/OpenAPI）
   - 脚本使用指南和集成示例
   - API 认证和权限控制（API Key 或 JWT）

5. **Excel 导入前预览**
   - 提供标准的 Excel 模板下载
   - Excel 导入前预览功能，确认无误后再提交
   - 差异比对和更新报告

6. **使用统计和分析**
   - 文案使用频率统计
   - 多语言覆盖率分析
   - 用户行为数据分析

### Vision (Future)

**未来愿景（梦想版本）：**

1. **智能重复文案检测**
   - 自动检测和标记重复的文案内容
   - 提供优化建议和智能合并方案

2. **按需下载语言包**
   - 支持按需下载特定语言的组合（如只下载中文和英文）
   - 减少带宽消耗和下载时间
   - 支持增量更新（只返回变更的文案）

3. **支持更多平台格式**
   - 扩展支持更多移动平台（如 Flutter、React Native）
   - 支持更多桌面平台（如 Electron、Tauri）
   - 支持自定义格式模板

4. **多租户和团队协作**
   - 支持多个团队独立使用
   - 权限管理和角色控制
   - 团队级别的文案隔离和共享

5. **自动化测试和持续集成**
    - 多语言文案的自动化测试
    - 与 CI/CD 流程深度集成
    - 质量监控和预警机制

## User Journeys

### 用户类型 1：产品经理小张（主要用户 - 成功路径）

**角色背景：**
- 负责产品界面的交互逻辑设计和文案撰写
- 工作在新功能设计完成和日常维护两个主要场景
- 需要管理所有界面的多语言文案
- 当前痛点：人工编写key耗时长、更新文案需要逐个通知开发人员、容易遗漏

**故事框架：**
- **姓名：** 产品经理小张
- **处境：** 新功能设计完成，需要为所有界面文案准备12种语言的翻译
- **目标：** 快速完成多语言文案准备，确保三端文案一致，不再需要逐个通知开发人员
- **障碍：** Excel中有100条文案，每条需要为12种语言准备翻译，人工处理需要1-2天
- **解决方案：** 使用 MultiLanguageManager 平台上传Excel，自动生成key，一键同步到三端

**用户旅程叙事：**

**开场场景（Opening Scene）：**
周一上午，小张刚完成新功能的交互设计，看着桌上的一叠多语言翻译Excel文件。100条文案，12种语言，这意味着1200个单元格需要核对。想到以往每次都要为每条文案手动编写key，然后分别联系iOS、Android、Web三端的开发人员，提醒他们更新对应的多语言文件，她就感到头痛。上次因为遗漏了某个文案的更新，导致三端文案不一致，被用户投诉了。

**上升动作（Rising Action）：**
这次，小张决定尝试团队新引入的 MultiLanguageManager 平台。她打开浏览器，登录系统，首先下载了标准的Excel模板。她将整理好的12种语言文案填入模板，点击"上传Excel"。

上传完成后，系统立即显示了预览界面：
- 检测到100条新文案
- 每条文案自动生成了唯一的UUID作为key
- 显示了各语言列的数据预览
- 标注了可能的空值或格式问题（有5条文案的芬兰语翻译缺失）

小张快速检查了预览内容，确认无误，点击"确认导入"。仅用了2分钟，100条文案就成功导入系统，生成了对应的iOS、Android、Web三种格式的多语言文件。

**高潮（Climax）：**
最关键的时刻来了。小张在Slack群组中发了一条消息："新功能多语言文案已更新到平台，各端可以拉取了。"

5分钟后，iOS开发人员回复："已拉取，验证通过，key都是新的UUID格式。"
Android开发人员回复："strings.xml已更新，打包测试中。"
Web开发人员回复："JSON文件已就位，准备部署。"

小张看着屏幕上的回复，意识到这次她没有逐个联系开发人员，没有发送Excel文件附件，没有担心漏掉某个文案。整个过程从上传到三端确认完成，只用了15分钟。

**结局（Resolution）：**
一个月后，小张使用平台已经习惯了新的工作流程。每次更新文案，她只需要：
1. 在Excel中修改需要更新的文案
2. 上传到平台，查看差异报告
3. 点击确认

开发团队会在收到通知后自动拉取更新。三端文案始终保持一致，再没有出现过因key不一致导致的用户投诉。小张每周节省了约8小时的多语言管理工作时间，她可以将这些时间投入到更有价值的产品思考和用户研究中。

**角色背景：**
- 负责各端多语言文案的集成和维护
- 需要将文案同步到对应项目的多语言文件中
- 当前痛点：手动复制粘贴key和value到对应的多语言文件中，效率极其低下

**故事框架：**
- **姓名：** iOS开发工程师小李
- **处境：** 新版本即将发布，产品经理更新了大量多语言文案
- **目标：** 快速同步最新的多语言文案到本地项目，确保打包时使用最新版本
- **障碍：** 以往需要手动从Excel中复制粘贴100+条文案到Localizable.strings文件，容易出错且耗时
- **解决方案：** 使用MultiLanguageManager提供的脚本，一键拉取并解压到项目目录

**用户旅程叙事：**

**开场场景（Opening Scene）：**
周五下午，小李正在准备新版本的发布工作。产品经理在Slack上通知："新功能多语言文案已更新到平台，各端可以拉取了。"

小李看着自己项目中庞大的Localizable.strings文件，心里想着：上次手动更新这个文件花了3个小时，还因为复制粘贴错误导致两个key的文案搞混了，结果被测试打回重测。这次，技术团队已经配置好了MultiLanguageManager的脚本，他决定试试。

**上升动作（Rising Action）：**
小李打开终端，切换到项目目录，运行命令：

```bash
./mlm-sync.sh --project-id "my-app" --platform ios --output ./Resources/Language
```

脚本立即开始工作：
- 连接到MultiLanguageManager API
- 获取最新的多语言文案数据
- 生成符合iOS标准的Localizable.strings文件
- 下载12种语言的文件压缩包
- 自动解压到指定的`./Resources/Language`目录

15秒后，脚本显示：

```
✓ Successfully connected to MultiLanguageManager
✓ Downloaded 12 language files (CN, DA, DE, EN, ES, FI, FR, IT, NL, NO, PL, SE)
✓ Extracted to ./Resources/Language/
✓ Total文案: 100条
✓ Processing time: 15.2 seconds
```

小李打开其中一个语言文件查看，格式完全正确，UUID作为key，所有文案都已准确映射。

**高潮（Climax）：**
小李将脚本集成到了项目的打包流程中。现在，每次打包前，Xcode会自动运行脚本拉取最新的多语言文案。这意味着：

- 不再需要手动更新多语言文件
- 打包时始终使用最新版本的文案
- 不会再出现因手动操作导致的格式错误或遗漏

小李运行了完整的打包测试，一切顺利。他看着测试应用中的多语言界面，文案都是最新的，三端的文案完全一致。

**结局（Resolution）：**
一周后，小李已经在开发流程中完全习惯了使用MultiLanguageManager。他不需要再担心产品经理更新文案后是否通知到位，因为脚本会在打包前自动拉取最新版本。

Android和Web开发的同事也配置了相应的脚本，三端团队现在使用统一的多语言文案源，再没有出现过因key不一致导致的问题。小李每周节省了约3小时的文案同步时间，他可以将这些时间投入到更有价值的开发和优化工作中。

### 用户类型 4：技术支持人员Sam（次要用户 - 故障排查）

**角色背景：**
- 负责处理用户使用MultiLanguageManager时遇到的问题
- 需要快速定位问题、提供解决方案

**故事框架：**
- **姓名：** 技术支持工程师Sam
- **处境：** 产品经理反映Excel上传失败
- **目标：** 快速定位问题原因，提供解决方案
- **障碍：** 需要查看日志、分析错误信息、指导用户正确操作
- **解决方案：** 使用平台的日志查询功能和诊断工具

**用户旅程叙事：**

**开场场景（Opening Scene）：**
周二上午，产品经理小张在Slack上@Sam："Excel上传失败，提示'文件格式错误'，我检查过了，格式应该是正确的，能帮忙看看吗？"

Sam立即响应："好的，请提供一下错误截图和Excel文件。"

**上升动作（Rising Action）：**
Sam开始排查问题：

1. **查看系统日志：**
   - 登录管理后台，进入日志查询页面
   - 筛选条件：用户=小张，操作=上传Excel，时间=今天上午
   - 找到错误日志：`FileParseError: Unsupported Excel format detected in file 'translations_v2.xlsx'`

2. **分析错误原因：**
   - 错误信息显示文件格式不支持
   - Sam猜测：小张可能使用了旧版.xls格式而非.xlsx格式
   - 或者Excel中包含了不支持的格式（如合并单元格、公式等）

3. **联系用户确认：**
   - Sam向小张询问："您的Excel文件是.xlsx格式吗？另外，文件中是否有合并单元格或公式？"

4. **诊断问题：**
   - Sam确认：问题出在公式上
   - 平台目前不支持包含公式的Excel文件
   - 需要小张将公式转换为静态值

**高潮（Climax）：**
Sam指导小张解决：

"Excel上传失败的原因是文件中包含公式。平台目前只支持静态文本值，不支持公式计算后的动态值。

解决方法：
1. 打开Excel文件
2. 选中包含公式的单元格
3. 复制这些单元格
4. 右键点击，选择'粘贴为值'
5. 保存文件后重新上传"

小张按照步骤操作后，Excel成功上传。她感谢道："谢谢！没想到是公式的问题，我之前都没注意到。"

**结局（Resolution）：**
一个月后，Sam处理了类似的几个问题。他发现，大部分问题都是由于：

- 包含公式或合并单元格
- 字符编码问题（非UTF-8）
- 空值处理（某些语言没有翻译）

     - 全面的单元测试，覆盖所有特殊字符和边界情况
     - 格式准确性验证（≥ 99.9%）
     - 提供下载预览功能，让用户验证生成的文件
     - 错误回滚机制

3. **API 安全风险**
   - **风险描述：** API Key 泄露、未授权访问、API 滥用
   - **缓解措施：**
     - API Key 加密存储
     - API 速率限制（Rate Limiting）
     - IP 白名单（仅允许公司内网访问）
     - 操作日志和审计

4. **并发访问风险**
   - **风险描述：** 多个用户同时操作导致数据冲突或性能问题
   - **缓解措施：**
     - 数据库事务处理
     - 乐观锁或悲观锁机制
     - 批量操作使用事务确保一致性
     - 性能监控和告警

5. **字符编码风险**
   - **风险描述：** 北欧语言特殊字符、emoji 等显示乱码或转换错误
   - **缓解措施：**
     - 严格的 UTF-8 编码验证
     - 特殊字符转义处理
     - 全面的字符编码测试
     - 文案预览功能

**业务风险：**

1. **用户采纳风险**
   - **风险描述：** 产品经理和开发人员不愿意使用新工具，继续使用旧的工作流程
   - **缓解措施：**
     - 提供清晰的使用文档和集成指南
     - MVP 阶段快速迭代，收集反馈
     - 提供技术支持和培训
     - 强调效率提升（8-16 倍效率提升）

2. **数据迁移风险**
   - **风险描述：** 从旧系统或现有文件迁移数据时出错或丢失
   - **缓解措施：**
     - 提供数据迁移工具或脚本
     - 迁移前备份现有数据
     - 迁移验证工具
     - 逐步迁移而非一次性迁移

**监控和告警：**

- **系统监控：** CPU、内存、磁盘使用率监控
- **API 性能监控：** 响应时间、成功率、错误率
- **使用统计：** 上传次数、下载次数、活跃项目数
- **告警配置：** 邮件、Slack 等告警机制

## Web App Specific Requirements

### Project-Type Overview

**MultiLanguageManager 是一个单页应用（SPA）的 Web 平台**，专注于内部开发团队的多语言文案管理。

**关键特性：**
- 产品经理通过浏览器上传 Excel、管理文案、查看进度
- 开发人员通过 API 和脚本拉取文件
- 系统管理员通过后台管理项目和监控

### Technical Architecture Considerations

**1. 浏览器支持：**
- **目标浏览器：** Chrome、Edge、Safari、Firefox（最新两个版本）
- **渐进增强：** 核心功能在所有浏览器中可用
- **浏览器降级：** 对于不支持某些特性的浏览器，提供降级方案

**2. 单页应用（SPA）架构：**
- **前端框架：** 建议 React 或 Vue.js（组件化、状态管理）
- **路由：** 使用客户端路由（React Router、Vue Router）
- **状态管理：** 需要管理项目、文案、批量操作状态

**3. 实时性：**
- **技术选择：** WebSocket 或 Server-Sent Events（SSE）
- **应用场景：**
  - Excel 上传进度
  - 批量操作进度（导入、导出、更新）
  - 实时日志查询
- **备选方案：** 轮询（如果 WebSocket 不可用）

**4. 响应式设计：**
- **断点：** Desktop（> 1024px）、Tablet（768-1024px）、Mobile（< 768px）
- **布局策略：**
  - Desktop：三栏布局（项目列表、文案列表、操作面板）
  - Tablet：两栏布局（项目列表 + 文案列表/操作面板）
  - Mobile：单栏布局，导航抽屉

**5. 性能目标：**
- **页面加载时间：** < 2 秒（首次内容绘制）
- **交互响应时间：** < 500ms（按钮点击、表单提交等）
- **文件上传：** 显示实时进度条
- **批量操作：** 显示实时进度条和预计完成时间

**6. 可访问性：**
- **目标标准：** WCAG AA 级别
- **关键要素：**
  - 键盘导航（Tab、Enter、Esc）
  - ARIA 标签（按钮、表单、进度条）
  - 色彩对比度（至少 4.5:1）
  - 屏幕阅读器兼容性

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP Approach:** 问题解决型 MVP - 专注于解决核心问题（三端 key 一致性），最小化功能
**Resource Requirements:** 2-3 人团队（1 全栈 + 1 后端 + 可选 1 UI/UX），开发周期 1-2 个月

### MVP Feature Set (Phase 1)

**Core User Journeys Supported:**
- ✅ 产品经理：Excel 上传 → 自动 key 生成 → 三端文件生成
- ✅ 开发人员：API 调用 → 脚本拉取 → 打包集成
- ✅ 系统管理员：项目创建 → 基础配置 → 监控

**Must-Have Capabilities:**
1. **Excel 上传和自动 key 生成**
   - 支持 12 种语言（CN、DA、DE、EN、ES、FI、FR、IT、NL、NO、PL、SE）
   - 自动生成唯一 UUID 作为 key
   - 支持按项目上传和管理多语言文案
   - 自动识别 Excel 中的新增和修改内容

2. **三端格式转换**
   - 支持 iOS（.strings）、Android（strings.xml）、Web（JSON）三种格式
   - 文件格式转换准确性 ≥ 99.9%
   - 下载的文件格式符合各平台标准

3. **按需下载**
   - 支持按项目、语言、平台筛选下载特定的多语言文件
   - API 响应时间（1000 条文案）< 10 秒
   - 同时支持脚本下载（通过 API）和手动下载（通过 Web 界面）

4. **项目基础管理**
   - 项目创建、编辑、删除
   - 项目配置（名称、描述、支持语言）
   - 标准的 Excel 模板下载

### Post-MVP Features

**Phase 2 (Post-MVP):**
- Web 界面单条编辑
- 文案搜索和筛选
- 变更历史和版本回滚
- Excel 导入前预览
- 批量操作（批量导入/导出/更新）
- 使用统计和分析
- API + 脚本支持（完整的 API 文档、脚本使用指南）
- API 认证和权限控制（API Key 或 JWT）
- 项目列表 API 端点（`GET /api/projects`）
- 错误处理和结构化响应

**Phase 3 (Expansion):**
- 智能重复文案检测
- 按需下载语言包
- 支持更多平台格式（Flutter、React Native、Electron）
- 多租户和团队协作
- 自动化测试和持续集成

### Risk Mitigation Strategy

**Technical Risks:** MVP 阶段简化实现，限制为标准 .xlsx 格式，要求用户提供静态值，全面的单元测试
**Market Risks:** 强调效率提升（8-16 倍），提供清晰的使用文档和集成指南，MVP 阶段快速迭代收集反馈
**Resource Risks:** 优先级排序，优先实现产品经理核心旅程，外包非核心功能，延后 Nice-to-Have 功能到 v2.0
